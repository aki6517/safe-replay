## 1. プロダクト概要

### 1.1 ビジョン

ADHD特性を持つフリーランス・個人事業主が、メール返信のストレスから解放されるAIアシスタント。

### 1.2 解決する課題

| 課題 | 詳細 |
|------|------|
| メールを開くこと自体が心理的に重い | ADHD特性による回避行動。未読が溜まり悪循環 |
| 返信を後回しにして信頼を失う | 「後で返す」が数日～数週間に。ビジネス上の損失 |
| 完璧主義で返信が書けない | 「60点の返信」を出せず、結果として0点（未返信） |
| 複数チャネルの管理が破綻する | Gmail・Chatwork・Slackを跨ぐ連絡の一元管理が困難 |

### 1.3 提供価値

1. **判断コスト削減**: AIがメールを自動分類（要返信/参考/不要）
2. **書き始めハードル低下**: AIが返信ドラフトを生成。ゼロから書く必要なし
3. **不安を煽らない通知**: ADHD向けに「語りかける」形式でLINE通知
4. **ワンタップ完了**: ボタン1つで返信送信。完了までの距離を最短に

### 1.4 ターゲットユーザー

- **属性**: ADHD診断済み or グレーゾーンの社会人
- **職業**: フリーランス・個人事業主（30代中心、デスクワーカー）
- **ツール**: Gmail + LINEを日常使い
- **課金感度**: 月額数千円の業務効率化ツールに投資する意識あり
- **初期規模**: 10人からスタート
- **対象地域**: 日本国内（UIは日本語のみ）

## 3. 非機能要件

### 3.1 可用性

| 項目 | 基準 |
|------|------|
| サービス稼働率 | 99%以上（月間ダウンタイム7時間以内） |
| 計画メンテナンス | 事前にLINE通知。深夜帯に実施 |

### 3.2 応答性

| 項目 | 基準 |
|------|------|
| ポーリング間隔 | 5分（Supabase Cron） |
| メール受信→通知 | 最大5分（ポーリング5分 + AI処理30秒） |
| ボタンタップ→返信送信 | 5秒以内 |

### 3.3 スケーラビリティ

| 項目 | 基準 |
|------|------|
| 初期ユーザー数 | 10人 |
| 現構成での上限 | 100人まで対応可能であること |
| ボトルネック | LINE API通知数 > Supabase Edge呼出数 > DB容量の順 |

### 3.4 セキュリティ

| 項目 | 対策 |
|------|------|
| 認証 | Gmail OAuth 2.0、LINE User ID、Service Role Key |
| 認可 | Row Level Security（全テーブル適用済み） |
| 通信暗号化 | Supabase標準TLS/HTTPS |
| 保存時暗号化 | Supabase PostgreSQL標準 |
| トークン管理 | Gmail/Chatwork/SlackトークンはユーザーごとにDB保存。アプリケーションレベルで暗号化（pgp_sym_encrypt or Supabase Vault） |
| PII取扱い | メール内容はDBに保存。90日後に自動削除（F-B6）。アカウント削除時は即時削除 |
| LIFF認証 | LIFF ID Token検証をサーバー側で実施。lineUserIdの詐称を防止 |
| 出力エスケープ | LIFF HTMLレンダリング時のユーザー入力は必ずサニタイズ（XSS防止） |
| APIキー | 環境変数（Supabase Secrets）で管理 |
| 監査ログ | メール送信アクション（message_actions）は不可変。90日保持期間を超えて1年間保持 |

### 3.5 データ保持

| データ | 保持期間 |
|--------|----------|
| メッセージ・ドラフト | 90日（以降自動削除） |
| 利用ログ（usage_logs） | 1年 |
| ユーザー設定 | アカウント削除まで |

### 3.6 監視・アラート

| 項目 | 方法 |
|------|------|
| ヘルスチェック | 15分間隔でDB/Redis疎通確認（GitHub Actions） |
| エラー通知 | APIトークン切れ・DB接続失敗→LINE緊急通知（運営者） |
| コスト監視 | usage_logsの月次集計で異常検知 |

### 3.7 災害復旧

| 項目 | 基準 |
|------|------|
| バックアップ | Supabase標準の日次自動バックアップ（Proプラン: Point-in-Time Recovery） |
| RTO（目標復旧時間） | 4時間以内 |
| RPO（目標復旧時点） | 24時間以内（最大1日分のデータ損失を許容） |
| 復旧手順 | Supabase管理画面からバックアップ復元。手順書を運用設計書に記載 |

## 4. AIモデル戦略

### 4.1 モデル構成

| 処理 | モデル | 理由 |
|------|--------|------|
| トリアージ（A/B/C分類） | gpt-5.4-nano | 分類タスク。速い・安い。精度十分 |
| ADHD向け柔軟化（要約変換） | gpt-5.4-nano | 要約＋言い換え。nanoで十分 |
| ドラフト生成 | gpt-5.4-mini | 文脈理解・ニュアンスが必要。品質重視 |
| カスタム修正 | gpt-5.4-mini | ユーザー指示の意図を正確に反映 |

※ 現行コードはgpt-4o-miniを使用。リリース時点で利用可能な最新モデルに更新する。nanoの前提料金: $0.10/1M入力, $0.40/1M出力。miniの前提料金: $0.40/1M入力, $1.60/1M出力。

### 4.2 スレッド文脈

- Gmailのthread_idで同一スレッドの直近10通を取得
- ドラフト生成時にgpt-5.4-miniの`threadHistory`パラメータとして渡す
- 「この会話の流れを踏まえて返信してください」というシステムプロンプト

### 4.3 コスト見積もり（1ユーザー/月）

前提: 1日平均10通、nano+mini構成

| 処理 | 入力トークン | 出力トークン | 回数/月 | 月額 |
|------|------------|------------|--------|------|
| トリアージ (nano) | ~800 | ~200 | 300 | ~$0.10 |
| 柔軟化 (nano) | ~1,000 | ~300 | 300 | ~$0.14 |
| ドラフト (mini) | ~2,000 | ~500 | 300 | ~$1.60 |
| 修正 (mini) | ~1,500 | ~500 | 30 | ~$0.14 |
| **合計** | | | | **~$1.98 ≈ ¥370/月** |

## 7. インフラ構成

### 7.1 現行アーキテクチャ

```
[ユーザーのGmail] → [Supabase Cron] → [Edge Functions (Deno)]
                                              ↓
                                    [OpenAI API (nano/mini)]
                                              ↓
                                    [Supabase PostgreSQL]
                                              ↓
                                    [LINE Messaging API]
                                              ↓
                                    [ユーザーのLINE]
```

### 7.2 サービス構成

| サービス | 用途 | プラン |

## 8. ユーザーストーリー

### US-01: 新規ユーザー登録

> ADHDフリーランスの田中さんは、SafeReplyのLINE Botを友だち追加する。
> ウェルカムメッセージとともにLIFF設定画面のURLが届く。
> ブラウザでGmailアカウントを連携し、設定を完了する。
> 以降、Gmailに届くメールが自動的にAI処理され、LINEに通知が届くようになる。

### US-02: 要返信メールの処理

> 田中さんのクライアントから「見積もりについて」のメールが届く。
> AIがType A（要返信）と判定し、返信ドラフトを生成。
> スレッドの過去のやり取りを踏まえた適切な返信案がLINEに届く。
> 田中さんは内容を確認し、[返信]ボタンを1タップ。メールが送信される。
> 「メールを開いて考えて書く」というストレスなく、30秒で完了。

### US-03: スヌーズの活用

> 田中さんは今すぐ返信できないメールを受信した。
> [後で対応]ボタンをタップし、2時間後を選択。
> 2時間後にLINEでリマインド通知が届き、そのまま返信操作ができる。

### US-04: VIPリストの設定

> 田中さんは重要クライアントからのメールを絶対に見逃したくない。
> LINE Botに「VIP追加 client@company.com」と送信。
> 以降、このアドレスからのメールはAIの分類に関係なく常にType A扱いで即通知される。

### US-05: Freeプランの上限到達

> 無料で使い始めた鈴木さんは、月5件の上限に到達。
> LINEに「今月の無料枠を使い切りました。Proプランにアップグレードすると無制限に使えます」と案内が届く。
> LIFFの決済画面からProプランに申し込む。

## 付録

### A. 用語定義

| 用語 | 定義 |
|------|------|
| Type A | 返信が必要なメッセージ。AIドラフト生成＋赤ヘッダー通知 |
| Type B | 参考/CC/共有メッセージ。緑ヘッダー通知。既読/確認メール送信可 |
| Type C | 不要なメッセージ（メルマガ等）。ログのみ、通知なし |
| LIFF | LINE Front-end Framework。LINEアプリ内ブラウザでWebページを表示 |
| トリアージ | メッセージの優先度を自動分類すること |
| スヌーズ | 一時的に通知を保留し、指定時間後にリマインドする機能 |
| VIPリスト | 重要送信者リスト。登録された送信者は常にType A扱い |
